## 八、錯誤處理（全域）

### 原則
* 不因為錯誤而中斷整體流程。每次遇到錯誤，先判斷、嘗試修復，再決定下一步。
* 錯誤是資訊，不是終點。

### 錯誤處理流程（每次遇到錯誤時執行）

1. **記錄錯誤** — 將錯誤訊息、發生位置、時間戳記錄到 History，並追加一筆錯誤事件到 agent_event_log.md
2. **判斷錯誤類型**
   * 格式錯誤（如 tool output 非字串）-> 修正輸出格式
   * 路徑不存在 -> 檢查路徑、建立目錄或改用已有檔案
   * 命令超時 -> 重試 1 次，仍超時則跳過
   * 權限不足 -> 調整權限或改用其他方式
   * 網路錯誤 -> 重試 1 次，仍失敗則記錄並跳過
   * 模型回應錯誤 -> 重新構造請求或改用備案
3. **嘗試修復**
   * 最多嘗試 2 次
   * 每次修復後驗證是否解決
4. **若修復失敗**
   * 標記 [!] 並記錄原因
   * 跳過該步驟，繼續執行 Next Step
   * 不要阻塞其他任務
5. **繼續執行** — 即使錯誤未完全解決，也繼續推進

### 常見錯誤對應策略

    錯誤類型                     應對方式
    400 tool output 非字串       將路徑/圖片輸出改為 stderr，stdout 僅輸出純文字
    FileNotFoundError            檢查路徑是否存在，建立或改用已有檔案
    timeout                      重試 1 次；仍超時 -> 記錄並跳過
    模型無回應                   重新發送請求，簡化 prompt
    權限不足                     改用有權限的方式，或記錄並跳過


## 九、agent_Tools_Record.md 記錄格式

### 原則
* agent_Tools_Record.md 只能追加，不可刪除或修改既有紀錄。
* 每次調用工具前或同一次工具流程中，必須先追加一筆工具紀錄。
* 每筆紀錄必須包含時間戳、工具名稱、用途摘要、所下指令，以及傳遞給工具的 JSON 參數。
* 若工具參數包含多行指令、路徑、權限設定、timeout、workdir、sandbox_permissions、justification 等，都必須保留在 JSON 中。
* 若 JSON 過長，可壓縮為單行 JSON；不可省略關鍵參數。

### 標準格式

    <timestamp> | tool_call_json=<compact-json>

### JSON 欄位建議

    {
      "tool": "工具名稱，例如 shell_command 或 read 或 write",
      "purpose": "本次工具調用目的摘要",
      "parameters": {
        "command": "實際執行的命令或操作內容",
        "workdir": "工作目錄，如適用",
        "timeout_ms": 30000,
        "sandbox_permissions": "require_escalated，如適用",
        "justification": "提權原因，如適用"
      }
    }

### 範例

    2026-05-25T09:37:43+08:00 | tool_call_json={"tool":"shell_command","purpose":"讀取 workflow 並檢查 AGENTS 規則","parameters":{"command":"Get-Content -LiteralPath agent_workflow.md -Raw","workdir":"D:\\Work\\25101\\ppt","timeout_ms":30000}}

### 錯誤與重試
* 若工具失敗，仍保留原工具紀錄，並在 agent_workflow_History.md 追加錯誤紀錄。
* 重試時必須追加新的 tool_call_json，不可覆蓋原紀錄。
* 若因格式錯誤導致 JSON 無法完整保存，下一筆紀錄需說明修正原因並改用有效 JSON。


## 十、多智能體協同工作規範

本節在前述各節的基礎上，定義多智能體協同工作的具體規則。所有多智能體操作（場景識別、角色分派、啟動子代理、評分、返工等）**必須同時遵守前述各節的通用規則**。

### 10.1 核心原則

1. **主代理（Orchestrator）只执行流程及子代理调配**
   * 主代理遵循本文件的模式規則（預設模式或持久模式）
   * 主代理不處理具體業務內容，只負責流程控制、狀態管理、子代理調度、生成控制文檔
   * 具體業務工作（編碼、設計、測試、文檔）由子代理完成

2. **所有控制文檔由主代理自動生成**
   * 場景規則文件：config/scene_rules.yaml
   * 子代理配置文件：.claude-agents/ 目錄下的多個 .md 文件
   * 任務詳情文件：tasks/ 目錄下的 .md 文件
   * 評分與返工記錄：tasks/reviews/ 目錄下的評分報告，以及 tasks/task-XXX.md 中的返工歷史

3. **子代理之間通過文件傳遞資訊**
   * 子代理只返回文件路徑給主代理，不返回大段內容
    * 主代理的上下文始終保持簡潔，適時使用系統內建 `compress` 工具管理上下文

4. **彈性角色分配**
   * 根據場景自動選擇合適的角色組合
   * 角色庫由主代理動態生成並維護

5. **獨立評分與返工**
   * 每個開發階段完成後，必須由 REVIEWER 子代理進行獨立客觀評分
   * 評分必須先填寫「評分檢查清單」（YES/NO），再給出總分
   * 評分低於 90 分時，該工作被視為不合格，必須「返工」
   * 返工視為新工作，重新規劃並執行，直到評分達到 90 分以上
   * 返工循環有上限（預設 5 次），超過則標記為「阻塞」請求人工介入

### 10.2 評分與返工詳細流程

#### 10.2.1 評分觸發時機

* 當一個任務的所有開發步驟被標記為完成，主代理必須啟動 REVIEWER 子代理。
* 不允許跳過評分直接標記為完成。

#### 10.2.2 評分者

* 必須使用專用的 REVIEWER 子代理（主代理在初始化時自動生成 .claude-agents/reviewer.md）。
* REVIEWER 不得是參與開發的任何子代理，以保證客觀性。
* 主代理不得自行評分，必須呼叫 REVIEWER。

#### 10.2.3 評分檢查清單（必須 YES/NO）

REVIEWER 在評分時，必須先針對以下四個項目逐一給出 YES 或 NO 的判定。這四項是評分的必要條件，任何一項為 NO 則總分不得超過 70 分。

    評分檢查清單（必須 YES/NO）
    - 是否可執行：YES / NO
    - 是否有錯誤：YES（代表沒有錯誤） / NO（代表有錯誤）
    - 是否滿足需求條列：YES / NO
    - 是否有測試或滿足審美：YES / NO

注意：「是否有錯誤」項目中，YES 表示沒有錯誤，NO 表示存在錯誤。REVIEWER 必須明確標註。

#### 10.2.4 評分標準與計算方式

REVIEWER 依據以下四項進行細項評分，每項 0-25 分，但必須參考檢查清單的結果：

* 完整性（25 分）：是否滿足用戶要求的所有功能點。若檢查清單中「是否滿足需求條列」為 NO，則此項最高得 10 分。
* 正確性（25 分）：邏輯、語法、設計是否正確無誤。若檢查清單中「是否有錯誤」為 NO（表示有錯誤），則此項最高得 10 分。
* 可維護性（25 分）：程式碼或文件是否清晰、有註解、易於修改。無強制約束，但低於 12 分需說明。
* 測試與驗證（25 分）：是否包含適當測試或驗證方法。若檢查清單中「是否有測試或滿足審美」為 NO，則此項為 0 分。此處「適當測試」指至少包含一個可自動執行的測試案例，覆蓋主要功能路徑。

額外條件：
* 若檢查清單中「是否可執行」為 NO，則總分直接為 0 分，不需進行細項評分。

總分 = 四項加總，範圍 0-100。**總分低於 90 分即為不合格**，必須返工。

#### 10.2.5 評分報告的存儲格式

REVIEWER 必須將評分結果寫入 tasks/reviews/ 目錄下的評分報告文件，命名規則為 review_<task-id>_<循環次數>.md，例如 review_TASK-001_2.md。

評分報告必須包含以下區塊：

    評分報告 for TASK-001 (第 2 次循環)

    評分時間: 2026-05-25T14:30:00+08:00
    評分者: reviewer-agent-002

    評分檢查清單（必須 YES/NO）:
    - 是否可執行: YES
    - 是否有錯誤: YES
    - 是否滿足需求條列: YES
    - 是否有測試或滿足審美: NO

    評分明細:
    - 完整性: 22/25
    - 正確性: 24/25
    - 可維護性: 20/25
    - 測試與驗證: 0/25 (原因: 檢查清單中「是否有測試」為 NO)

    總分: 66/100
    結果: 不合格 (低於 90)

    缺失項目與改進建議:
    1. 缺少單元測試 (檢查清單「是否有測試」為 NO)
    2. 程式碼註解不足

    具體建議:
    - 新增 test_xxx.py 檔案，覆蓋主要功能
    - 為所有公開方法補上 docstring

REVIEWER 完成後只返回評分報告的路徑，例如 tasks/reviews/review_TASK-001_2.md。

#### 10.2.6 主代理處理評分結果

主代理讀取評分報告路徑後，判斷總分是否 >= 90：

* 若 >= 90：將任務狀態設為「已完成」，在 tasks/task-XXX.md 中記錄最終評分報告路徑，並繼續下一個任務。
* 若 < 90：將任務狀態設為「退回」，返工次數加 1，記錄退回原因（評分報告路徑），然後啟動返工流程。

#### 10.2.7 返工流程

1. **重新規劃**：主代理呼叫 PLANNER 子代理（resume），傳入原始需求、前次計劃、評分報告，要求根據報告中的建議調整計劃，輸出新計劃文檔。
2. **重新執行**：主代理按照新計劃，啟動開發子代理（優先 resume 原開發代理）執行任務。
3. **重新評分**：任務完成後，再次啟動 REVIEWER 子代理進行評分，生成新的評分報告（循環次數遞增）。
4. **循環終止條件**：
   * 評分達到 90 分以上 -> 任務完成。
   * 返工次數達到 5 次仍未達標 -> 主代理將任務狀態設為「阻塞」，記錄「需人工介入」，停止自動返工。
5. **返工記錄**：每次返工必須在 tasks/task-XXX.md 的「返工歷史」區塊追加一條記錄。

#### 10.2.8 任務狀態與評分相關的欄位（tasks/task-XXX.md）

每個任務詳情文件必須包含以下區塊：

    # TASK-001 開發登入元件

    ## Status
    退回

    ## Current Score
    66

    ## Rework Count
    2

    ## Review Reports
    - tasks/reviews/review_TASK-001_1.md (score 75, 退回)
    - tasks/reviews/review_TASK-001_2.md (score 66, 退回)

    ## Rework History
    - 第1次返工: 評分75，檢查清單: 可執行=YES, 無錯誤=YES, 滿足需求=YES, 有測試=NO
    - 第2次返工: 評分66，檢查清單: 可執行=YES, 無錯誤=YES, 滿足需求=YES, 有測試=NO

    ## 最終評分（完成時）
    （待填寫）

### 10.4 多智能體工作流程（按模式分解）

#### 在「預設模式」下

主代理可一次執行多個步驟，流程為：

1. 讀取用戶需求，保存到 tasks/requirements.md
2. 生成控制文檔（若無）
3. 場景識別與角色分派
4. 啟動 PLANNER 子代理生成計劃
5. 循環執行各任務：
   a. 啟動開發子代理執行任務
   b. 開發完成後，啟動 REVIEWER 子代理評分（必須先填寫檢查清單）
   c. 若評分 >= 90，標記完成；若 < 90，返工次數 +1，根據評分報告重新規劃，回到步驟 a（最多 5 次，超過則阻塞）
6. 所有任務完成後生成總結報告

#### 在「持久模式」下

主代理必須將上述流程拆解為多個最小步驟，每次只執行一個步驟，更新 agent_workflow.md 後停止。例如：

* 步驟1：讀取需求，保存文件，更新 Next Step =「生成控制文檔」
* 步驟2：生成控制文檔，更新 Next Step =「場景識別」
* ...
* 步驟N：開發完成後，啟動 REVIEWER 評分，更新 Next Step =「檢查評分結果」
* 步驟N+1：讀取評分報告，判斷分數；若不合格且未達上限，更新 Next Step =「重新規劃（返工）」；若合格則標記完成

## 十一、事件日誌、狀態與檢視層規範

本節定義三種核心文件的分工與操作規則，適用於所有模式（預設模式與持久模式）。

### 11.1 三種文件的定義

1. **agent_event_log.md（不可變事件日誌）**
   * 這是一個只能追加（append-only）的事件日誌文件。
   * 每當系統發生任何重要事件（任務開始、步驟完成、子代理啟動、評分、返工、錯誤、模式切換等），必須按時間順序追加一條事件記錄。
   * 已追加的事件記錄**絕不可修改或刪除**。
   * 事件記錄格式為：
        [時間戳] | [事件類型] | [事件詳情（簡要）]
   * 範例：
        2026-05-25T10:00:00+08:00 | TASK_START | 任務 TASK-001 開始
        2026-05-25T10:15:00+08:00 | AGENT_INVOKE | 啟動 PLANNER 子代理，計劃文件 tasks/plan.md
        2026-05-25T10:45:00+08:00 | REVIEW | 任務 TASK-001 評分 75，不合格
        2026-05-25T10:50:00+08:00 | REWORK | 任務 TASK-001 第 1 次返工

2. **agent_state.md（派生狀態）**
   * 這是一個由系統自動計算或彙總的狀態文件，反映當前系統的整體快照。
   * 它**不是**獨立輸入，而是根據 agent_event_log.md 和 tasks/*.md 等數據派生（derived）而來。
   * 主代理在每次更新後可以重新生成或更新 agent_state.md，但不可直接手動編輯（除非初始化）。
   * 內容應包含：
        - 當前活動任務列表與狀態
        - 整體進度百分比
        - 當前模式（預設/持久）
        - 最近一次錯誤（若有）
        - 活躍的子代理清單
   * 範例內容：
        ## 系統狀態（派生，最後更新 2026-05-25T11:00:00）
        - 當前模式: 預設模式
        - 整體進度: 45%
        - 活動任務: TASK-001 (進行中), TASK-002 (待辦)
        - 最近錯誤: 無
        - 活躍子代理: planner-001, reviewer-002

3. **agent_workflow.md（視圖層 / 工作流視圖）**
   * 這是人類或上層系統主要查看的視圖文件，用於了解當前工作流進度。
   * 它應包含「模式」、「Current Step」、「Next Step」等原有欄位，以及對事件日誌和狀態的引用（可選）。
   * 主代理每次更新 workflow 時，應同時：
        - 追加一筆事件到 agent_event_log.md
        - 重新生成 agent_state.md（可選，但建議每次重大變化後更新）。重大變化包括：任務狀態變更、評分完成、返工開始或結束、模式切換、錯誤發生
        - 更新 agent_workflow.md 中的 Current Step 和 Next Step 等欄位
   * 三者關係：agent_event_log.md 是歷史真相，agent_state.md 是當前快照，agent_workflow.md 是指導下一步的視圖。

### 11.2 操作規則（強制）

1. **寫入順序**：當發生狀態變更時，必須依序執行：
   a. 追加事件到 agent_event_log.md（不可變）
   b. 更新 agent_state.md（重新派生）
   c. 更新 agent_workflow.md（更新 Current Step / Next Step）

2. **讀取優先級**：
   * 若要了解歷史脈絡：讀取 agent_event_log.md
   * 若要了解當前整體狀態：讀取 agent_state.md
   * 若要決定下一步行動：讀取 agent_workflow.md 中的 Next Step

3. **錯誤恢復**：若 agent_state.md 或 agent_workflow.md 損壞或丟失，可以根據 agent_event_log.md 重新派生它們。重建步驟如下：
   1. 掃描 agent_event_log.md 中所有事件
   2. 提取最新的事件類型與時間戳
   3. 根據事件序列重建當前狀態（哪些任務活躍、當前模式、最新評分等）
   4. 寫入 agent_state.md 和 agent_workflow.md
   因此 agent_event_log.md 是最終的可靠來源。

4. **初始化**：首次運行時，主代理必須建立空的事件日誌、初始狀態文件、初始工作流文件。範例初始化內容：

   agent_event_log.md:
        2026-05-25T00:00:00+08:00 | SYSTEM_START | 系統初始化，事件日誌建立

   agent_state.md:
        ## 系統狀態（派生）
        - 當前模式: 預設模式
        - 整體進度: 0%
        - 活動任務: 無
        - 最近錯誤: 無
        - 活躍子代理: 無

   agent_workflow.md:
        # Agent Workflow
        ## 模式
        預設模式
        ## Current Step
        - [ ] 尚未設定
        ## Next Step
        - 尚未設定

### 11.3 與前述各節的整合

* 第四節「共通規則」中原本要求「必須讀取 agent_workflow.md」和「必須更新 agent_workflow.md」，現擴充為：每次讀取 agent_workflow.md 前，可先參考 agent_state.md 取得狀態快照；每次更新 agent_workflow.md 時，必須同時追加事件到 agent_event_log.md 並考慮更新 agent_state.md。
* 第八節「錯誤處理」中，當發生錯誤時，除了記錄到 History，也必須追加錯誤事件到 agent_event_log.md。
* 第九節「agent_Tools_Record.md」記錄工具調用，這是另一個日誌；agent_event_log.md 記錄更高層級的業務事件（任務、評分、返工等），兩者相輔相成，不可互相取代。

### 11.4 範例工作流更新片段（持久模式下）

當主代理完成一個最小步驟（例如啟動 PLANNER 子代理）後，應執行：

1. 追加事件到 agent_event_log.md：
   2026-05-25T10:05:00+08:00 | AGENT_INVOKE | 啟動 PLANNER，計劃輸出 tasks/plan.md

2. 重新生成 agent_state.md（更新活躍子代理清單，進度可能不變）：
   ## 系統狀態（派生）
   - 活躍子代理: planner-001

3. 更新 agent_workflow.md：
   ## Current Step
   - [x] 啟動 PLANNER 子代理
   ## Next Step
   - 檢查 PLANNER 輸出並讀取計劃文檔路徑

### 11.5 注意事項

* agent_event_log.md 不可變的性質意味著你不能回刪或修改舊事件。若發現錯誤事件，應追加一條「勘誤」事件來說明。
* agent_state.md 是派生數據，主代理可在任何時候重新計算並覆蓋它，不必保留歷史。
* agent_workflow.md 仍然遵循持久模式下的「每次只做一個最小步驟」的規則，Current Step 和 Next Step 必須精確反映單步進度。
